# SEO & AEO Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the site's SEO/AEO gaps in one pass — shared metadata/JSON-LD infrastructure, a sitemap and robots.txt, homepage metadata (currently missing entirely), per-entity structured data (tournaments, matches, players, exchange listings, TV videos), dynamic OG images, and AEO extras (`llms.txt`, a homepage FAQ with `FAQPage` schema).

**Architecture:** A small `lib/seo/` module holds pure, testable functions — a `buildMetadata()` helper that every page's `generateMetadata`/`metadata` export calls instead of hand-rolling OpenGraph objects, and one JSON-LD builder function per schema.org type. Pages stay thin: they fetch their own data (already true throughout this codebase) and pass it through the shared builders. Next.js file conventions (`robots.ts`, `sitemap.ts`, `opengraph-image.tsx`) handle the parts Next has first-class support for.

**Tech Stack:** Next.js 14 App Router metadata APIs, `next/og` `ImageResponse`, Supabase (via `createAdminClient()` for the sitemap, which runs unauthenticated), Vitest for the pure-function layer (this repo only runs `.test.ts` files, not `.tsx` — component/page files aren't unit tested anywhere in this codebase, and this plan follows that existing convention).

## Global Constraints

- `SITE_URL` = `process.env.NEXT_PUBLIC_SITE_URL ?? 'https://sentinelx.gg'` — never hardcode a domain elsewhere; import from `lib/seo/site.ts`.
- `SITE_NAME` is the full form `'SentinelX Esports'`; `'SentinelX'` is only a shorthand for the root title template.
- Currency in structured data is always `'NGN'`.
- The sitemap's Supabase queries **must** use `createAdminClient()` (`lib/supabase/admin.ts`), never `createClient()` — sitemap generation has no authenticated user, so RLS would silently return zero rows.
- Canonical URLs for filterable listing pages (`/tournaments`, `/players`, `/exchange`) are always the bare path with no query string. This is implemented by omission (the page's `generateMetadata` never appends `searchParams` to the `path` it passes to `buildMetadata`) — **not** a generic query-stripping utility. If pagination (`?page=`) is added to any of these pages in the future, that page's canonical logic must be revisited; don't let a future `?page=2` silently canonicalize to page 1.
- `next/font` fonts are not usable inside `ImageResponse`. **Deviation found during implementation:** the plan's approach of fetching `app/fonts/GeistVF.woff` directly doesn't work at all — `GeistVF`/`Rajdhani` are *variable* fonts, and `@vercel/og`'s Satori renderer can't parse the `fvar` (variation axis) table in a variable font; it throws (`Cannot read properties of undefined (reading '256')`) regardless of platform, not just in this dev environment. `lib/og/template.tsx` loads no custom font at all and relies on Satori's built-in default — correctness over brand-exact typography. A related fallout: Satori's default font has no glyph for `₦`, so the two OG images that show a price (`tournaments/[slug]` and `exchange/[id]`) render `NGN 13,000` instead of `formatNaira()`'s `₦13,000`.
- All five `opengraph-image.tsx` files run on `export const runtime = 'edge'`. Node runtime hits a separate, Windows-specific `@vercel/og` bug (`TypeError: Invalid URL` from `fileURLToPath` while loading its own WASM) that crashes the build; edge runtime uses a different code path that doesn't have it, and it's also how Vercel's own OG-image examples run in production.
- `buildMetadata` (`lib/seo/metadata.ts`) omits `openGraph.images`/`twitter.images` entirely when no explicit `image` is given — it does **not** default to `DEFAULT_OG_IMAGE` internally. This is deliberate: Next only auto-fills a route's `opengraph-image.tsx` into its metadata when that segment's own `generateMetadata` doesn't return an `openGraph` object with `images` set; it does **not** cascade a file-convention image down from an ancestor segment (a child segment's `openGraph` object fully replaces the parent's rather than deep-merging). Concretely: the 4 pages with their own same-segment `opengraph-image.tsx` (`tournaments/[slug]`, `players/[username]`, `matches/[id]`, `exchange/[id]`) omit `image` when there's no banner/photo and let Next fill in their own dynamic image. Every other page calling `buildMetadata` (home, the 6 static/listing pages, the bracket page, `/tv`) has no segment-local image file and must pass `image: DEFAULT_OG_IMAGE` explicitly, or it silently loses its OG image entirely.
- **Deviation from the approved spec, flagged here:** the spec named a hand-authored static `public/og-default.png`. This plan instead builds `app/opengraph-image.tsx` (Next's root-level file convention) using the same shared OG template as the per-route dynamic images — it renders the identical "logo + tagline on brand background" the spec described, but is generated from the real brand template instead of being a separately hand-produced asset that would need to be kept in sync by hand. `DEFAULT_OG_IMAGE` points at the URL Next serves that file at.
- `VideoObject` schema is only emitted for videos with a real `uploadDate` (curated videos always have `published_at`; match replays/finals always have `completed_at` since only completed matches with a replay reach those sections). The live-stream hero on `/tv` has no fixed upload date and does not get `VideoObject` schema — this is a Google Search Central requirement (`uploadDate` is a required property), not scope creep.

```ts
import type { Metadata } from 'next'
import { SITE_URL, SITE_NAME, DEFAULT_OG_IMAGE } from './site'

export type BuildMetadataInput = {
  title: string
  description: string
  /** Route path with no query string, e.g. '/tournaments/dls-26-championship'. */
  path: string
  image?: string
  type?: 'website' | 'article'
}

export function buildMetadata({ title, description, path, image, type = 'website' }: BuildMetadataInput): Metadata {
  const url = `${SITE_URL}${path}`
  const ogImage = image ?? DEFAULT_OG_IMAGE
  return {
    title,
    description,
    alternates: { canonical: url },
    openGraph: {
      title,
      description,
      url,
      siteName: SITE_NAME,
      type,
      images: [ogImage],
    },
    twitter: {
      card: 'summary_large_image',
      title,
      description,
      images: [ogImage],
    },
  }
}
```

```ts
import type { MetadataRoute } from 'next'
import { SITE_URL } from './site'

export function staticSitemapEntries(): MetadataRoute.Sitemap {
  const paths = ['/', '/tournaments', '/players', '/rankings', '/hall-of-fame', '/tv', '/exchange', '/community']
  return paths.map((path) => ({ url: `${SITE_URL}${path === '/' ? '/' : path}` }))
}

export function tournamentSitemapEntry(row: { slug: string; updated_at: string | null }): MetadataRoute.Sitemap[number] {
  return {
    url: `${SITE_URL}/tournaments/${row.slug}`,
    lastModified: row.updated_at ?? undefined,
    changeFrequency: 'daily',
    priority: 0.8,
  }
}

export function playerSitemapEntry(row: { username: string; updated_at: string | null }): MetadataRoute.Sitemap[number] {
  return {
    url: `${SITE_URL}/players/${row.username}`,
    lastModified: row.updated_at ?? undefined,
    changeFrequency: 'weekly',
    priority: 0.5,
  }
}

export function matchSitemapEntry(row: { id: string; completed_at: string | null }): MetadataRoute.Sitemap[number] {
  return {
    url: `${SITE_URL}/matches/${row.id}`,
    lastModified: row.completed_at ?? undefined,
    changeFrequency: 'monthly',
    priority: 0.3,
  }
}

export function listingSitemapEntry(row: { id: string; updated_at: string | null }): MetadataRoute.Sitemap[number] {
  return {
    url: `${SITE_URL}/exchange/${row.id}`,
    lastModified: row.updated_at ?? undefined,
    changeFrequency: 'daily',
    priority: 0.4,
  }
}
```

- [ ] **Step 6: Create `app/sitemap.ts`, wiring the mappers to Supabase via `createAdminClient()`**

```ts
import type { MetadataRoute } from 'next'
import { createAdminClient } from '@/lib/supabase/admin'
import {
  staticSitemapEntries,
  tournamentSitemapEntry,
  playerSitemapEntry,
  matchSitemapEntry,
  listingSitemapEntry,
} from '@/lib/seo/sitemap-entries'

export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  const supabase = createAdminClient()

  const [{ data: tournaments }, { data: players }, { data: matches }, { data: listings }] = await Promise.all([
    supabase.from('tournaments').select('slug, updated_at').neq('status', 'draft'),
    supabase.from('profiles').select('username, updated_at').gt('total_matches', 0).not('username', 'is', null),
    supabase.from('matches').select('id, completed_at').eq('status', 'completed'),
    supabase.from('marketplace_listings').select('id, updated_at').eq('status', 'active'),
  ])

  return [
    ...staticSitemapEntries(),
    ...(tournaments ?? []).map(tournamentSitemapEntry),
    ...(players ?? [])
      .filter((p): p is { username: string; updated_at: string | null } => p.username != null)
      .map(playerSitemapEntry),
    ...(matches ?? []).map(matchSitemapEntry),
    ...(listings ?? []).map(listingSitemapEntry),
  ]
}
```

```ts
import { SITE_URL, SITE_NAME } from '../site'

export type TournamentEventInput = {
  title: string
  slug: string
  description: string | null
  status: string
  tournamentStart: string | null
  tournamentEnd: string | null
  registrationFee: number
}

export function buildTournamentJsonLd(t: TournamentEventInput) {
  const url = `${SITE_URL}/tournaments/${t.slug}`
  return {
    '@context': 'https://schema.org',
    '@type': 'SportsEvent',
    name: t.title,
    description: t.description ?? `${t.title} on Sentinel X.`,
    url,
    ...(t.tournamentStart ? { startDate: t.tournamentStart } : {}),
    ...(t.tournamentEnd ? { endDate: t.tournamentEnd } : {}),
    eventStatus: 'https://schema.org/EventScheduled',
    eventAttendanceMode: 'https://schema.org/OnlineEventAttendanceMode',
    location: { '@type': 'VirtualLocation', url },
    organizer: { '@type': 'Organization', name: SITE_NAME, url: SITE_URL },
    offers: {
      '@type': 'Offer',
      price: t.registrationFee,
      priceCurrency: 'NGN',
      url,
      availability: 'https://schema.org/InStock',
    },
  }
}

export type MatchEventInput = {
  id: string
  playerAName: string
  playerBName: string
  status: string
  scoreA: number | null
  scoreB: number | null
  tournamentTitle: string | null
  tournamentSlug: string | null
}

export function buildMatchJsonLd(m: MatchEventInput) {
  const url = `${SITE_URL}/matches/${m.id}`
  const name = `${m.playerAName} vs ${m.playerBName}`
  const completed = m.status === 'completed' && m.scoreA != null && m.scoreB != null
  return {
    '@context': 'https://schema.org',
    '@type': 'SportsEvent',
    name,
    url,
    competitor: [
      { '@type': 'Person', name: m.playerAName },
      { '@type': 'Person', name: m.playerBName },
    ],
    ...(m.tournamentTitle && m.tournamentSlug
      ? {
          superEvent: {
            '@type': 'SportsEvent',
            name: m.tournamentTitle,
            url: `${SITE_URL}/tournaments/${m.tournamentSlug}`,
          },
        }
      : {}),
    description: completed ? `${name} — final score ${m.scoreA}–${m.scoreB}.` : name,
  }
}
```

```tsx
  return r?.display_name ?? r?.username ?? 'TBD'
}

export default async function Image({ params }: { params: { id: string } }) {
  const supabase = createClient()
  const { data: m } = await supabase
    .from('matches')
    .select(
      'score_a, score_b, status, player_a:profiles!matches_player_a_id_fkey(username, display_name), player_b:profiles!matches_player_b_id_fkey(username, display_name)',
    )
    .eq('id', params.id)
    .maybeSingle()

  const a = nameOf(m?.player_a ?? null)
  const b = nameOf(m?.player_b ?? null)
  const scored = m?.status === 'completed' && m.score_a != null && m.score_b != null

  return renderOgImage({
    title: `${a} vs ${b}`,
    subtitle: scored ? `${m!.score_a} – ${m!.score_b}` : undefined,
  })
}
```

```ts
export type FaqItem = { question: string; answer: string }

export function buildFaqJsonLd(items: FaqItem[]) {
  return {
    '@context': 'https://schema.org',
    '@type': 'FAQPage',
    mainEntity: items.map((item) => ({
      '@type': 'Question',
      name: item.question,
      acceptedAnswer: { '@type': 'Answer', text: item.answer },
    })),
  }
}
```
