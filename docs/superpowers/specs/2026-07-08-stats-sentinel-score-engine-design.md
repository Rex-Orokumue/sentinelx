# Stats & Sentinel Score Engine — Design Spec

**Task:** #10a (backend half of v2.0 #10 "Player profiles")
**Date:** 2026-07-08
**Status:** Approved design → ready for implementation plan

---

## 1. Why this exists

The `profiles` table has every field a player profile needs — `wins`, `losses`,
`total_matches`, `goals_scored`, `goals_conceded`, `total_titles`,
`sentinel_score`, `sentinel_tier` — but **nothing writes to them.**

When an admin confirms a match result (`confirmResult` in
`lib/matches/verify-actions.ts`), the code updates `group_memberships` standings
and advances the bracket, but it never touches the profile aggregates, never
writes a `sentinel_score_events` row, and never recomputes `sentinel_score`. So
in production every player still shows **0 wins / score 70**, and the
already-built **Rankings** and **Hall of Fame** pages are effectively empty
(Rankings filters on `total_matches >= 1`, which is never true).

This engine makes that data real. Building it first (before the profile page,
10b) means the page lands on real numbers, and it lights up Rankings and Hall of
Fame with **no changes to those pages.**

---

## 2. Core principle — derive, don't increment

Two values are **derived projections**, each recomputed from scratch from its own
source of truth. The `profiles` columns are **caches**, refreshed after every
change, never the source of truth.

| Projection | Source of truth | Recompute |
|---|---|---|
| **Aggregates** (`wins`, `losses`, `total_matches`, `goals_scored`, `goals_conceded`, `total_titles`) | the `matches` table (`completed` only) + `getChampion` for titles | `computeAggregates(...)` |
| **Sentinel Score** (`sentinel_score`) | the `sentinel_score_events` log | `computeScore(events) = clamp(70 + Σ points_delta, 0, 100)` |

`sentinel_tier` is a DB **generated column** — we only ever write
`sentinel_score`; the tier follows automatically.

Because both are full recomputes-from-source, every operation is **idempotent**
and every bug is recoverable: fix the code, re-run the recompute, get the correct
number back. Incremental deltas on `profiles` are explicitly rejected — a single
wrong delta would silently corrupt a score with no recovery path.

### The log has two kinds of rows

- **Auto match-derived** — `event_type ∈ {match_completed, win_no_dispute}`.
  Generated by this engine from a completed match, tagged with `match_id`.
  **Regenerated** (delete + reinsert) whenever that match's result changes.
- **Authored** (future) — `dispute_lost`, `rating_received`, `admin_flag_*`,
  `no_show`, `rage_quit`. Written directly by their own admin/rating UIs (which
  don't exist yet). **Never auto-deleted** by this engine, even if they carry a
  `match_id`.

The discriminator for "safe to regenerate" is **`event_type` membership in the
auto set**, never `match_id != null` — because an authored event (e.g. a
`dispute_lost` tied to a match) can also carry a `match_id`.

`computeScore` simply **sums the stored `points_delta`** of all a player's
events. It needs no delta lookup table — the delta rules live only where events
are *created* (`events.ts`). This also makes future star ratings trivial: a 5★
(+2) and a 1★ (−2) are both `rating_received` rows with different stored deltas.

## 3. Schema change — migration `006_win_no_dispute_event.sql`

The current `event_type` CHECK constraint omits `win_no_dispute`, so the +1 win
bonus from the CLAUDE.md scoring model has no home. Rather than fold a variable
delta into `match_completed` (2 for losers, 3 for winners — a dishonest,
hard-to-audit ledger), extend the enum so each row keeps **one clear reason and a
fixed delta per type**.

```sql
ALTER TABLE public.sentinel_score_events
  DROP CONSTRAINT sentinel_score_events_event_type_check,
  ADD CONSTRAINT sentinel_score_events_event_type_check
    CHECK (event_type IN (
      'match_completed', 'win_no_dispute', 'no_show', 'rage_quit',
      'dispute_lost', 'rating_received', 'admin_flag_conduct', 'admin_flag_cheat'
    ));
```

`event_type` is a `text` column with a CHECK (not a Postgres enum type), so the
generated TypeScript type is already `string` — **no `lib/supabase/types.ts`
regeneration needed.** The migration must be applied to the live Supabase project
as part of implementation.

## 4. Module layout — `lib/scoring/`

Follows the repo convention: pure logic in small modules, each with a colocated
`.test.ts`; impure orchestration kept thin.

### 4.1 `score.ts` (pure)

```ts
export const BASE_SCORE = 70
export function computeScore(events: { points_delta: number }[]): number
// clamp(BASE_SCORE + Σ points_delta, 0, 100)
```

### 4.2 `events.ts` (pure)

```ts
export const AUTO_MATCH_EVENT_TYPES = ['match_completed', 'win_no_dispute'] as const
export const MATCH_COMPLETED_DELTA = 2
export const WIN_DELTA = 1

export interface NewMatchEvent {
  player_id: string
  match_id: string
  event_type: 'match_completed' | 'win_no_dispute'
  points_delta: number
  note: null
}

export function matchEventsFor(match: {
  id: string
  player_a_id: string | null
  player_b_id: string | null
  score_a: number | null
  score_b: number | null
  status: string
}): NewMatchEvent[]
```

Rules:
- Only `status === 'completed'` produces events. Any other status → `[]`.
- Each **present** player (non-null id) gets `match_completed` (+2).
- The **winner** (decisive, `score_a !== score_b`) gets `win_no_dispute` (+1).
- A **draw** (group play) → both get `match_completed`, no win bonus.
- A **bye** (`player_b_id` null / `status='bye'`) → `[]` (the player auto-advanced
  without playing; a bye is not a completed match).

### 4.3 `stats.ts` (pure)

```ts
export interface Aggregates {
  total_matches: number
  wins: number
  losses: number
  goals_scored: number
  goals_conceded: number
  total_titles: number
}

export function computeAggregates(
  playerId: string,
  completedMatches: { player_a_id, player_b_id, score_a, score_b }[],
  titlesWon: number,
): Aggregates
```

For each completed match, normalize to the player's own side (are they A or B),
then: `total_matches` = count; `wins`/`losses` by score comparison (draws count
as neither — `profiles` has no draws column, by design); `goals_scored` /
`goals_conceded` summed from the player's perspective. `titlesWon` is passed in
(computed in the orchestration layer via `getChampion`).

### 4.4 `apply.ts` (impure orchestration — service-role admin client only)

All writes touch other users' rows and the append-only log, so they **must** go
through `createAdminClient()` (service role, bypasses RLS). Never the user
client. `confirmResult` already uses the admin client.

```ts
// Regenerate one match's auto-events. Returns affected player ids. No refresh.
regenerateMatchEvents(admin, match): Promise<string[]>
//   DELETE ... WHERE match_id = match.id
//              AND event_type IN AUTO_MATCH_EVENT_TYPES
//   INSERT matchEventsFor(match)   (if any)

// Recompute + write both caches for one player.
refreshPlayer(admin, playerId): Promise<void>
//   titlesWon  ← count of completed finals where getChampion === playerId
//   aggregates ← computeAggregates(playerId, player's completed matches, titlesWon)
//   score      ← computeScore(player's full event log)
//   UPDATE profiles SET wins,losses,total_matches,goals_scored,
//          goals_conceded,total_titles,sentinel_score   (tier auto-derives)

// Used by confirm/dispute hooks.
syncMatchEvents(admin, matchId): Promise<void>
//   fetch match → regenerateMatchEvents → refreshPlayer for each of the
//   two players (both refreshed even when the match left 'completed', so
//   their totals drop symmetrically)

// Full rebuild — the admin recompute button and recovery tool.
recomputeAllScoring(admin): Promise<{ players: number }>
//   DELETE ... WHERE event_type IN AUTO_MATCH_EVENT_TYPES   (all auto rows)
//   for each completed match: INSERT matchEventsFor(match)
//   for each profile: refreshPlayer
//   (authored events are preserved; returns count for the UI)
```

## 5. Hooks into the result flow

In `lib/matches/verify-actions.ts`, reusing the existing admin client:

- **`confirmResult`** — after the match status flips to `completed` (and
  alongside the existing standings/advancement logic), `await syncMatchEvents(admin, id)`.
- **`disputeResult`** — after status → `disputed`, `await syncMatchEvents(admin, id)`.
  The match is no longer `completed`, so its auto-events are deleted and none
  reinserted; both players refresh downward.

Re-confirming a disputed match with new scores just re-runs `syncMatchEvents` —
delete, reinsert fresh, recompute. No diffing, no stale rows.

Titles update naturally: confirming a `final` refreshes the two finalists, and
`refreshPlayer` recomputes `total_titles` from completed finals via `getChampion`
(reused from `lib/tournaments/bracket`, never reimplemented).

## 6. Admin recompute button

- **Location:** `/admin` dashboard — a "Recompute all scores & stats" control.
- **Gate:** `requireAdmin` (Samuel only — heavier than a moderator action).
- **Action:** `recomputeAllAction` calls `recomputeAllScoring(admin)`.
- **Confirmation:** a small client component (`RecomputeButton`) with a two-step
  in-app confirm — the button reveals "Yes, recompute all players" + "Cancel".
  **No native `window.confirm`** (blocking, not mobile-first, not testable).
- **Feedback:** on success, "Recomputed N players." on the page.

This is the 11pm-during-a-live-tournament recovery path: a button, no shell.

## 7. Testing

Pure functions carry the coverage, TDD, per repo convention:

- **`score.test.ts`** — empty log → 70; sums deltas; clamps at 0 (heavy
  negatives) and 100 (heavy positives); mixed positives/negatives.
- **`events.test.ts`** — completed decisive → winner `[completed+2, win+1]`,
  loser `[completed+2]`; draw → both `[completed+2]`, no win; bye and
  non-completed statuses → `[]`; null player handling.
- **`stats.test.ts`** — player as A and as B; wins/losses/goals from the
  player's perspective; draws count as neither; titles passed through; excludes
  non-completed matches.

`apply.ts` orchestration is thin; verified by running the recompute against seed
data and confirming Rankings/Hall of Fame populate.

---

## 8. Scope boundaries

**In 10a:** the migration; `lib/scoring/` pure modules + tests;
confirm/dispute hooks; the admin recompute button; making Rankings & Hall of
Fame show real data.

**Not in 10a (no trigger surface exists yet):** event-*writing* for
`rating_received` (needs the opponent-rating UI), `admin_flag_*` (needs the admin
flag UI), `no_show` / `rage_quit` (match status enum can't represent them);
suspension on cheat flag; the player profile page itself (**10b**). `computeScore`
still understands all of these — only the writing is deferred.

**Verification bonus:** after 10a lands and the recompute runs, `/rankings` and
`/hall-of-fame` should populate with no code changes to those pages — a good
end-to-end signal that the engine works.
